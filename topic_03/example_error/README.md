# example_error

Приклади на те, як помилки в обробниках доходять до клієнта і навіщо потрібен один спільний error middleware.

## Файли прикладів

- [`ex01.ts`](./ex01.ts) — звернення до даних за неіснуючим `id` з примусовим `!`: демо «сирих» винятків без error middleware — процес падає.

- [`ex02.ts`](./ex02.ts) — `throw` у роуті при відсутності запису + єдиний error middleware: одна точка, що перетворює виняток на JSON і статус.

- [`ex03.ts`](./ex03.ts) — власний клас помилки з HTTP-статусом: розрізнення клієнтських (`400`, `404`) та несподіваних (`500`) у форматі відповіді.

- [`ex04.ts`](./ex04.ts) — акцент на самому error middleware: структуроване логування (`message`, `stack`, `url`, `method`, час) і уніфікований JSON клієнту (у файлі немає окремих роутів — лише цей шар).
